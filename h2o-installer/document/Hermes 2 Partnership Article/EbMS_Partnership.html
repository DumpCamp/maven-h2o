1.1 Partnership ID

Description The unique identifier of the ebMS 2.0 partnership in local Hermes 2 machine.
The value of this field has no restriction but RECOMMENDED to be an identifiers between the sending party and receiving party.

It is mandatory and it's maximum length of this field is 255.
Top

1.2 CPA ID

Description The CPA (Collaboration Protocol Agreement) id. It is a string that identifies the parameters governing the exchange of messages between the parties. The recipient of a message MUST be able to resolve the CPAId to an individual set of parameters, taking into account the sender of the message.

Simply said, it is just an arbitary and mandatory string that can be used to identify sender and receiver.
Top

1.3 Service

Description In Hermes 2, the Service parameter for Sending Partnership and Receiving Partnership has to be HTTP URL (Universal Resource Locator) even though ebMS specification allows a Service value that is not HTTP URL. The Service has different meanings in Send Partnership and Receive Partnership:
  1. For Sending Partnership, the Service acts as the endpoint URL of the sending Hermes to which the acknowledgement is sent by the receiver. That is, this should be the endpoint URL of the sending Hermes for receiving ebMS messages.
  2. For Receiving Partnership, since the Service is a filtering criterion for the incoming messages, it should be the endpoint URL of the sending Hermes.
Top

1.4 Action

Description It identifies a process within a Service that processes the Message. Action SHALL be unique within the Service in which it is defined. The value of the Action element is specified by the designer of the service. It is mandatory and it's maximum length of this field is 255.
Top

1.5 Disabled

Description The boolean options indicates whether the partnership is disabled or not.
Disabled partnership does not deliver / receive any outgoing message / incoming respectively.

It is RECOMMENDEDto set it to false.
Options [ true = disabled ], [ false = enabled ]
Top

1.6 Transport Endpoint

Description The endpoint URL of the receiving message gateway.

If the receiving message gateway is Hermes 2, the endpoint URL is formatted as http://<RECEIVER HOST>:<PORT>/corvus/httpd/ebms/inbound.

Otherwise, if the receiving message gateway is SMTP gateway, the endpoint URL is formatted as mailto:EMAIL_ADDRESS.

It is mandatory and the format MUST be a HTTP/HTTPS URL/EMAIL ADDRESS.
Top

1.7 Hostname Verified in SSL?

Description The boolean flag indicates whether HTTP SSL/TLS protocol is used and required verifying the receipient hostname.

It is relevant if HTTPS transport protocol is used under the Transport Endpoint
Options [ true = hostname verified using SSL , false = none ]
Top

1.8 Sync Reply Mode

Description Indicates whether the receiver should reply it's ebMS message in same HTTP/HTTPS connection the sender uses for delivery.

This parameter is only applied to HTTP/HTTPS protocol and applicable for sending partnership.
Options [ mshSignalsOnly = synchronous reply ], [ none = asynchronous reply ]

An example of synchronous reply from the receipient:
The reply ebMS message is included in the HTTP/SOAP Response when synchronous reply is applied.


An example of asynchronous reply from the receipient:
The reply ebMS message will be delivered through another HTTP/SOAP connection from the receipient to the sender.

1.9 Acknowledgment Requested

Description Indicates whether the receiver is requested to send an ebMS acknowledgment back to the sender.

Both sender and receiver MUST enable this (by setting to always) for interoperability of this features, otherwise the receiver returns the negative acknowledgment.

An acknowledgment is a kind of ebMS Message which has <acknowledgment> element. It is an receipt that reciever party sends back acknowledgment to the sender party after receiving ordinary ebMS message w ith/with-out payload.

The mode of acknowledgment sending back depends on the value Sync Reply Mode, if Sync Reply Mode is enabled, the acknowledgment will send back immediately in the same HTTP connection. otherwise, the acknolwedgment will be put in outgoing buffer queue and wait indefinitely before send back to sender if the receiver is using Hermes 2.

It is RECOMMENDED to set this parameter to always for reliable messaging.
Options [ always = acknowledgment requested ], [ none = acknowledgment does not request ]
Top

1.10 Acknowledgment Signing Requested

Description Indicates whether the receiver MUST sign the ebMS acknowledgment digitally using it's private key before delivering back to the sender.

Both sender and receiver MUST enable this (by setting to true) for interoperability of this features, otherwise the receiver returns the negative acknowledgment.

The format of the private key SHOULD BE in PKCS12 and the signatures created is conformed to W3C XML Signatures Specification [XMLDsig].

The sender partnerships MUST request acknowledgment from receiver for running this features properly. Also, It is required to provide the public certificates for verifying the signatures in the acknowledgment.
Param(s) dependencies [ Acknowledgment Requested = always ],
[ Certificate For Verification REQUIRED ]
Options [ true = acknowledgment MUST be digitally signed ],
[ false = acknolwedgment MUST not be digitally signed ]
Top

1.11 Duplicate Elimination

Description Indicates whether the receiver eliminate/ignore duplicate received message.

Both sender and receiver MUST enable this (by setting to always) for interoperability of this features, otherwise the receiver returns the negative acknowledgment.
Options [ alawys = ignores / eliminates duplicate message. ],
[ never = delivers duplicate message. ]
Top

1.12 Message Order

Description Indicates whether the receiver MUST RECEIVES the ebMS messages in the order of time requesting Hermes 2 through web service in sender system.

Both sender and receiver MUST enable this (by setting to Guaranteed) for interoperability of this features, otherwise the receiver returns the negative acknowledgment.

For example, if the external application / client has requested sender Hermes 2 to deliver ten ebMS message with each request is later than the previous one by a second. The expected result in receiver system using Message Order SHOULD be the order of received message SHOULD be the same in the application / client requested to Hermes 2.
Param(s) dependencies [ Sync Reply Mode = none ],
[ Acknowledgment Requested = always ],
[ Duplicate Elimination = always ]
Options [ Guaranteed = The receiver receives ebMS messages in the order of sending. ],
[ NotGuaranteed = The receiver receives in best effort behavior. ]
Top

1.13 Signing Required

Description Indicates whether the sender MUST sign the normal/payload ebMS Message digitally using it's private key.

Both sender and receiver MUST enable this (by setting to true) for interoperability of this features, otherwise the receiver returns the negative acknowledgment.

The format of the private key SHOULD BE in PKCS12 and the signatures created is conformed to W3C XML Signatures Specification [XMLDsig].
Options [ true = The outgoing normal/payload ebMS message MUST be digitally signed. ],
[ false = The outgoing ebMS message does not require to sign digitally. ]
Top

1.14 Encryption Required

Description Indicates whether the sender MUST encrypt the normal/payload ebMS Message using the corresponding public certifcates defined in Cert for Encryption.

It is applicable only when using SMTP protocol.

The encryption method is based on S/MIME standard.
Param(s) dependencies [ Transport Endpoint = started with mailto: ],
[ Sync Reply Mode = none ],
[ Certificate For Encryption REQUIRED ]
Options [ true = The outgoing normal/payload ebMS message MUST be encrypted. ],
[ false = The outgoing ebMS message does not require to encrypt. ]
Top

1.15 Encryption Cert

Description The certificates file for encrypting the outgoing ebMS message through SMTP protocol by using the public key generated by the receiving party. Hence, the certificate should contains the public key generated by receiving party.

(For Recevier Side)In order to generate certificate for sending party, you should generate the certificate by using the keystore in EbMS plugin.
Keystore path: hermes2_path/plugins/hk.hku.cecid.ebms/security

The format of the certificates MUST be PKCS12. See Encryption Required for details.
Top

1.16 Maximum Retries

Description The maximum number of retry that the Sender can attempt to deliver the ebMS Message.

Hermes 2 tries to deliver the ebMS Message again under the features of reliable messaging until meeting the maximum number of retries.

Each retry will be executed after a interval defined in retry interval from the last delivery attempt.
Param(s) dependencies [ Acknowledgment Requested = always ]
Options [ recommended range = 1-10 ]
Top

1.17 Retry Interval

Description The interval in millesecond between consecutive retry in delivery.
Param(s) dependencies [ Acknowledgment Requested = always ]
Options [ recommended range = 30000 - 300000 ]
Top

1.18 Verification Cert

Description The certificates file for verifying the received ebMS message with digital signatures. The certificate should containing a public key generated by the sending party.

The format of the certificates MUST be PKCS12 and it should be the certificate for the subject, not the certificate of its certificate of its certificate authority.

(For Sender Side)In order to generate certificate for receiving party to perform verification, the certificate should be generated by using the keystore in EbMS plugin.
Keystore path: hermes2_path/plugins/hk.hku.cecid.ebms/security

See Signing Required for details.
